# Braincode 长期任务与运行时编排改造方案

> 本文档用于指导 Codex 在 Braincode 项目中分阶段实施改造。
>
> EEG_agent 教学项目会被放入 Braincode 项目中作为设计参考。实施时应吸收其设计思想和状态机，不应直接把教学代码整块复制到生产代码中，也不应让 Braincode 的生产模块依赖参考目录。

## 1. 改造目标

将 Braincode 从以交互式会话为中心的编码助手，逐步增强为能够可靠执行长期任务、后台任务和定时任务的 Agent Runtime，同时保持现有异步架构、TUI、CLI、Remote、MCP、权限、Sandbox、Team、Worktree 和 Session 能力兼容。

本轮改造重点吸收 EEG_agent 的以下优势：

1. 持久化任务 DAG、依赖校验、claim 和崩溃恢复。
2. 统一的 LLM 重试、退避、模型降级和上下文恢复状态机。
3. Cron 定时触发和持久化调度。
4. 普通工具和子 Agent 的后台执行能力。
5. 可修改输入、输出和上下文的结构化 HookResult。
6. 基于运行时状态签名的 Prompt section 组装与缓存。
7. 任务、调度、后台执行和运行时通知的一体化可观测性。

## 2. 非目标

本次改造不应执行以下工作：

- 不把 Braincode 重写成 EEG_agent 的同步循环架构。
- 不引入以线程为主的第二套 Agent 调度器。
- 不替换 Braincode 已有的 `Tool`、`ToolRegistry`、`LLMClient`、`ConversationManager` 等抽象。
- 不使用 EEG_agent 的全局 `TOOLS`、`TOOL_HANDLERS` 或全局模型客户端模式。
- 不用字符数估算替代 Braincode 现有 token usage anchor 和 compact 机制。
- 不用字符串 denylist 替代 Braincode 的 PermissionChecker 和 OS Sandbox。
- 不用教学项目中的 Mock MCP 替换 Braincode 真实 MCP 实现。
- 不在第一阶段大规模修改 TUI 或 Remote UI。
- 不让生产代码导入 EEG_agent 参考目录中的任何模块。

参考时遵循以下原则：

- 先理解行为，再按 Braincode 的异步、类型化和模块化风格重新实现。
- 优先复用 Braincode 已有模型与服务，不建立重复数据源。
- 所有持久状态都要有 schema/version，能够安全读取旧数据。
- 所有新增行为必须有自动化测试。
- 参考目录只读，不修改、不导入、不打包。

## 5. 全局工程原则

### 5.1 保持 asyncio 单一并发模型

- 新增服务使用 `asyncio.Task`、`asyncio.Queue`、async context manager。
- 不新增常驻 Python 线程轮询器。
- Cron 调度器应运行在现有应用事件循环中。
- TUI、CLI、Remote 应调用同一套服务，不分别实现运行时逻辑。

### 5.2 SQLite 优先于多 JSON 文件

Braincode 已支持 Team、Remote 和潜在多进程后端，任务 claim 不能只依赖进程内锁。

建议使用 Python 标准库 `sqlite3`：

- 不新增第三方数据库依赖。
- 开启 WAL。
- claim 使用事务和条件更新。
- 数据库默认位于项目的 `.braincode/runtime.db`。
- 为 schema 增加版本表和轻量迁移机制。

若实施者认为 SQLite 与当前架构冲突，必须先说明原因，再选择替代方案；不得直接采用无原子保证的普通 JSON claim。

### 5.3 向后兼容

- 保持现有工具名、命令名和 AgentTool 行为可用。
- 新功能默认不改变普通交互会话行为。
- Cron 默认关闭，只有存在配置或用户创建任务时启动。
- 原 `TaskManager` 公共接口应保留，内部可委托给 `JobManager`。
- 原 Team shared task 数据应继续可读，必要时做一次性迁移或兼容适配。

### 5.4 安全边界

- 后台执行不等于绕过权限。
- 权限判断必须在任务入队前完成，执行前还要验证持久化权限上下文是否合法。
- 禁止将交互式 `ask` 权限请求挂在无人值守 Cron 中无限等待。
- 定时任务默认使用 `default` 权限模式；需要确认时应失败并产生可读通知。
- 后台 Bash 必须继续使用 Braincode 的 Sandbox 和 Path/Rule 权限体系。

## 7. Phase 1：持久化 Job 核心

### 目标

实现统一、持久、可恢复、支持依赖的 Job 数据层，但暂不接入 Cron。

### 7.1 Job 模型

建议定义枚举：

```python
class JobKind(StrEnum):
    AGENT = "agent"
    TOOL = "tool"
    PROMPT = "prompt"
    TEAM = "team"

class JobStatus(StrEnum):
    PENDING = "pending"
    RUNNING = "running"
    COMPLETED = "completed"
    FAILED = "failed"
    CANCELLED = "cancelled"
    BLOCKED = "blocked"
```

建议字段：

```text
id
kind
name
description
status
payload_json
result_text
error_text
owner
team_name
worktree_path
created_at
updated_at
started_at
finished_at
lease_until
attempts
max_attempts
priority
parent_job_id
schedule_id
```

依赖关系单独存储为 `job_dependencies(job_id, depends_on_job_id)`。

### 7.2 Store 接口

```python
class JobStore(Protocol):
    def create(self, spec: JobSpec) -> Job: ...
    def get(self, job_id: str) -> Job | None: ...
    def list(self, query: JobQuery | None = None) -> list[Job]: ...
    def claim(self, job_id: str, owner: str, lease_seconds: int) -> Job | None: ...
    def heartbeat(self, job_id: str, owner: str, lease_seconds: int) -> bool: ...
    def complete(self, job_id: str, owner: str, result: str) -> Job: ...
    def fail(self, job_id: str, owner: str, error: str) -> Job: ...
    def cancel(self, job_id: str) -> Job: ...
    def recover_expired(self, now: datetime) -> list[Job]: ...
```

### 7.3 必须实现的约束

- [ ] 依赖任务不存在时拒绝创建。
- [ ] 重复依赖拒绝或去重并记录。
- [ ] 自依赖拒绝。
- [ ] 创建和更新依赖时进行环检测。
- [ ] 只有全部依赖完成的任务可以 claim。
- [ ] claim 必须是数据库原子操作。
- [ ] complete/fail 必须验证 owner。
- [ ] lease 过期的 running job 可恢复为 pending 或 failed。
- [ ] 超过 `max_attempts` 后不得无限恢复。
- [ ] 所有时间以 UTC 存储。

### 7.4 测试

- [ ] Job CRUD。
- [ ] 不存在依赖。
- [ ] 自依赖。
- [ ] 两节点和多节点环。
- [ ] 并发 claim 只有一个成功。
- [ ] owner 不匹配不能完成。
- [ ] lease 续约。
- [ ] 进程重启后的过期任务恢复。
- [ ] schema migration。

### 验收条件

- JobStore 可以独立通过测试。
- 尚未改变现有 TaskManager 对外行为。
- 数据库损坏或 schema 不支持时能给出明确错误，不静默丢任务。

## 9. Phase 3：LLM RecoveryController

### 目标

在 `LLMClient` 与 `Agent` 之间建立统一恢复策略。

### 9.1 错误模型

扩展或重构现有异常，使其至少包含：

```python
class LLMError(Exception):
    status_code: int | None
    retryable: bool
    retry_after: float | None
    provider_name: str
```

建议分类：

- AuthenticationError：不可重试。
- InvalidRequestError：通常不可重试。
- RateLimitError：可重试，尊重 Retry-After。
- OverloadedError：可重试，可触发 provider fallback。
- NetworkError：可重试。
- ContextLimitError：执行 compact 后最多重试一次。
- StreamInterruptedError：根据流式阶段决定是否重试。

### 9.2 RecoveryState

```text
attempt
rate_limit_attempts
overload_attempts
network_attempts
context_recovery_attempted
output_token_escalated
continuation_count
current_provider_index
stream_started
any_output_emitted
tool_call_completed
tool_execution_started
```

### 9.3 安全重试边界

只有满足以下条件才允许自动重放整个模型请求：

- 尚未执行任何工具。
- 尚未提交有副作用的外部操作。
- 尚未向 Conversation 写入不可撤销的新状态。

若已经产生部分文本但没有工具调用，可选择：

- 将部分文本丢弃并重试；或
- 保存部分文本并请求模型续写。

必须选择一种明确策略并编写测试，不能隐式混用。

### 9.4 退避策略

```text
delay = min(base * 2^attempt, max_delay) + jitter
```

- Retry-After 优先。
- 默认最大重试次数有限。
- fallback provider/model 必须来自配置，不硬编码。
- 认证失败不得切换到共享同一无效凭证的 provider 后无限重试。

### 9.5 接入位置

- [ ] `Agent.run()` 主模型调用。
- [ ] `Agent.run_to_completion()`。
- [ ] Memory recall/extraction/consolidation 的模型调用。
- [ ] Remote 和非交互 CLI 不再分别实现恢复逻辑。
- [ ] UI 能显示 RetryEvent 的原因、次数和 provider 切换。

### 测试

- [ ] 429 + Retry-After。
- [ ] 529 连续过载。
- [ ] 网络抖动后成功。
- [ ] fallback model/provider。
- [ ] 上下文超限后 compact 一次。
- [ ] max_tokens escalation 和 continuation 上限。
- [ ] 已执行工具后不重放。
- [ ] 流中断前后不同安全边界。

### 验收条件

- 所有模型协议使用同一恢复语义。
- 不会产生重复工具副作用。
- 每次恢复操作都有可观测事件。

## 10. Phase 4：Cron Scheduler 与后台工具

### 目标

在 JobManager 上实现持久定时任务和普通工具后台执行。

### 10.1 Schedule 模型

建议字段：

```text
id
name
cron_expression
timezone
job_kind
payload_json
enabled
next_run_at
last_run_at
misfire_policy
overlap_policy
created_at
updated_at
```

策略枚举：

```text
misfire_policy: skip | run_once
overlap_policy: skip | coalesce | parallel
```

默认建议：

```text
misfire_policy = skip
overlap_policy = coalesce
timezone = local configured timezone
```

### 10.2 调度原则

- Scheduler 到期时只创建 Job，不直接进入当前 Conversation。
- 定时 prompt 使用独立 Session。
- 调度器重启后根据策略处理 missed runs。
- 用唯一 fire key 防止同一时间点重复触发。
- 调度器不得绕过 Job 依赖、权限和 max attempts。

### 10.3 后台工具

建议支持：

```python
AgentTool(..., run_in_background=True)
Bash(..., run_in_background=True)
```

后台工具 Job 的 payload 必须固化：

- 工具名。
- 已校验参数。
- cwd/worktree。
- permission mode。
- sandbox 配置摘要。
- 创建者和 parent job。

权限要求：

- 入队前执行权限判断。
- 执行前重新验证安全上下文。
- `ask` 决策不能在无人值守 Job 中阻塞。
- 权限不足时 Job 失败并通知，不自动升级到 bypass。

### 10.4 工具与命令

建议增加或扩展：

- `JobCreate`
- `JobGet`
- `JobList`
- `JobCancel`
- `CronCreate`
- `CronList`
- `CronDelete`

TUI 命令建议：

- `/jobs`
- `/cron`

先实现核心服务和工具，再做复杂 UI。

### 测试

- [ ] Cron 表达式校验。
- [ ] 时区转换。
- [ ] missed run 策略。
- [ ] overlap 策略。
- [ ] 重启后不重复触发。
- [ ] 后台 Bash 完成、失败、超时和取消。
- [ ] 后台 Bash 继续使用 Sandbox。
- [ ] 后台输出过大时落盘并只注入摘要。

### 验收条件

- Cron、后台 Agent、后台工具共用 JobManager。
- 没有第二套任务状态系统。
- 定时任务可跨重启继续工作。

## 11. Phase 5：结构化 HookResult

### 目标

保留 Braincode 现有 Hook action/condition 能力，同时允许 Hook 修改输入、输出和上下文。

### 11.1 数据模型

建议新增：

```python
@dataclass
class HookResult:
    outcome: str = "continue"
    message: str = ""
    reject_reason: str = ""
    prevent_continuation: bool = False
    updated_args: dict[str, Any] | None = None
    updated_output: str | None = None
    additional_context: str = ""
```

### 11.2 合并优先级

多个 Hook 命中时使用固定规则：

```text
reject/cancel
  > prevent_continuation
  > updated_args/updated_output
  > additional_context
  > notification only
```

- updated args 按 Hook 执行顺序合并。
- reject 后停止执行后续会产生副作用的 Hook。
- async hook 不允许同步修改当前工具参数；异步 Hook 只能通知或为后续轮次注入上下文。
- 旧 `reject: true` 配置映射到新的 reject outcome。

### 11.3 接入事件

- [x] user_prompt_submit。
- [x] session_start/session_end。
- [x] turn_start/turn_end。
- [x] pre_send/post_receive。
- [x] pre_tool_use/post_tool_use。
- [x] permission_request。
- [x] stop。

### 11.4 测试

- [x] 修改工具参数。
- [x] 两个 Hook 连续修改同一字段。
- [x] reject 优先级。
- [x] post hook 修改工具输出。
- [x] additional context 注入下一轮。
- [x] async hook 不修改当前调用。
- [x] 旧配置兼容。

### 验收条件

- Hook 可以成为稳定扩展点。
- PermissionChecker 仍是最终安全裁决者。
- Hook 不能通过 updated args 绕过已经完成的权限检查；修改参数后必须重新校验权限。

## 12. Phase 6：PromptStateRegistry

### 目标

让运行时子系统以统一方式向 system prompt 提供状态，并保持 prompt cache 友好。

### 12.1 Provider 接口

```python
class PromptStateProvider(Protocol):
    @property
    def name(self) -> str: ...

    @property
    def priority(self) -> int: ...

    def signature(self) -> Hashable: ...

    def render(self) -> str: ...
```

首批 Provider：

- Job 状态。
- Cron 状态。
- Team 状态。
- Worktree 状态。
- MCP 状态。
- Skills 目录。
- Memory 摘要。
- Recovery/provider 状态。

### 12.2 缓存与稳定性

- section 按 priority 和 name 稳定排序。
- signature 未变化时复用已渲染 section。
- 空状态不注入。
- 快速变化的进度数据不要进入稳定 system prompt。
- 当前时间等易变字段放入环境消息或按需 reminder。
- 记录 section cache hit/miss 和重建原因，但默认日志保持简洁。

### 12.3 与 Provider prompt cache 的关系

Braincode 已在模型客户端中设置 prompt cache 断点。此阶段的目标不是替代上游缓存，而是使 system prompt 字节前缀更稳定，从而提高上游缓存命中率。

### 测试

- [ ] 相同 signature 不重建。
- [ ] 单个子系统变化只重建对应 section。
- [ ] section 顺序稳定。
- [ ] 空状态不注入。
- [ ] volatile 状态不会破坏稳定 section。
- [ ] coordinator mode 行为不回归。

### 验收条件

- Prompt 组装逻辑不再直接了解每个运行时子系统的内部结构。
- 新增状态源只需注册 Provider。
- Anthropic prompt cache 命中不低于改造前基线。

## 13. Phase 7：统一初始化、UI 与可观测性

### 目标

减少 TUI、CLI、Remote 初始化重复，并让新运行时能力可见、可操作。

### 13.1 RuntimeContainer

建议抽取公共运行时容器或工厂：

```python
class RuntimeContainer:
    client
    agent
    registry
    job_manager
    scheduler
    team_manager
    worktree_manager
    mcp_manager
    hook_engine
    prompt_state_registry
    recovery_controller
```

由统一 builder 根据配置创建，供以下入口使用：

- Textual TUI。
- `braincode -p`。
- RemoteServer。

### 13.2 可观测事件

建议统一 RuntimeEvent：

```text
job_created
job_started
job_progress
job_completed
job_failed
job_cancelled
schedule_fired
retry_started
provider_switched
hook_modified_input
hook_rejected
```

TUI、CLI stream-json、Remote WebSocket 应从同一事件源转换输出。

### 13.3 UI 最小能力

- 查看运行中和最近完成 Job。
- 查看 Cron 列表和下次运行时间。
- 取消 Job。
- 查看 retry/provider switch 提示。
- 点击或命令读取持久化大输出。

### 验收条件

- 三种入口使用相同 RuntimeContainer。
- 新服务只初始化一次。
- 关闭应用时 Scheduler、MCP、Job 和 Team 都能有序停止。

## 14. 配置建议

在现有配置模型中按需加入：

```yaml
runtime:
  database: .braincode/runtime.db
  recover_jobs_on_start: true
  job_lease_seconds: 120
  max_job_attempts: 3

recovery:
  max_retries: 6
  base_delay_seconds: 0.5
  max_delay_seconds: 32
  max_output_continuations: 3
  fallback_providers: []

scheduler:
  enabled: false
  timezone: Asia/Shanghai
  poll_interval_seconds: 1
  default_misfire_policy: skip
  default_overlap_policy: coalesce
```

要求：

- 配置验证放在系统边界。
- 未配置时使用安全默认值。
- 不应因为 Scheduler 未使用而创建无意义后台循环。
- 敏感凭证仍由现有 ProviderConfig 管理。

## 16. 每阶段执行规则

Codex 每次只实施一个 Phase 或其中一个可独立验证的子阶段。

每次实施必须遵循：

1. 先读取当前实现和相关测试。
2. 检查工作区是否已有用户修改，避免覆盖。
3. 写出本次最小改动边界。
4. 优先新增兼容层，再迁移调用方。
5. 不进行无关重构或全项目格式化。
6. 为新增行为编写测试。
7. 先运行定向测试，再运行完整测试。
8. 报告改动、测试结果、已知限制和下一阶段入口。
9. 某阶段验收未通过时，不继续叠加下一阶段。

推荐实施顺序：

```text

  ↓

  ↓

  ↓
Phase 3 RecoveryController
  ↓
Phase 4 Cron + Background Tool
  ↓
Phase 5 HookResult
  ↓

  ↓
```
